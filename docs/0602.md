## Session 3 — 기본 보안 설정 점검 및 적용

**목표**: OWASP Top 10 기준 응답 헤더, CORS, 환경변수, 관리자 API 접근 제어.

### 3-A. 응답 보안 헤더 추가 (Gateway)

**현황**: Gateway `SecurityConfig`에 응답 헤더 설정 없음.

**적용할 헤더**:

| 헤더 | 값 | 목적 |
|---|---|---|
| `X-Content-Type-Options` | `nosniff` | MIME 스니핑 방지 |
| `X-Frame-Options` | `DENY` | Clickjacking 방지 |
| `X-XSS-Protection` | `1; mode=block` | 레거시 브라우저 XSS 필터 |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Referrer 정보 누출 제한 |
| `Permissions-Policy` | `geolocation=(), microphone=()` | 불필요한 브라우저 API 차단 |

**구현 위치**: `Gateway SecurityConfig` 또는 전용 `SecurityHeaderFilter` (GlobalFilter).

### 3-B. CORS 설정 확인 및 강화

**확인 사항**:
- Gateway `application.yml`의 `globalcors` 설정 여부
- `allowedOrigins`가 `*`(와일드카드)인지, 특정 origin인지
- credentials(쿠키/Authorization) 허용 여부

**목표 설정**:
```yaml
spring:
  cloud:
    gateway:
      globalcors:
        corsConfigurations:
          '[/**]':
            allowedOrigins:
              - "http://localhost:5173"
              # 프로덕션: 실제 도메인으로 교체
            allowedMethods: [GET, POST, PUT, DELETE, OPTIONS]
            allowedHeaders: [Authorization, Content-Type, Idempotency-Key, X-Queue-Token, X-Captcha-Token]
            allowCredentials: true
            maxAge: 3600
```

### 3-C. Actuator 노출 범위 제한

**현황**: JWT 필터 면제 경로에 `/actuator/**` 포함 → 외부에서 인증 없이 접근 가능.

**수정**:
- 각 서비스 `application.yml`: `management.endpoints.web.exposure.include=health,prometheus,info,circuitbreakers`
- `/actuator/shutdown`, `/actuator/env`, `/actuator/beans` 비노출 확인
- Gateway: `/actuator/prometheus` 경로는 내부망(Prometheus 스크레이핑) 전용 — IP 제한 또는 basicauth 검토

### 3-D. 관리자 API 접근 제어 확인

**대상**: `GET /api/admin/dlt-alerts` (train, payment, notification 3개 서비스)

**확인 사항**:
```java
// 각 AdminController에 @PreAuthorize 또는 역할 체크 있는지 확인
// 없으면 추가:
if (!"ADMIN".equals(userRole)) throw new BusinessException(ErrorCode.ACCESS_DENIED);
```

### 3-E. 환경변수 하드코딩 전수조사

**점검 대상**:
```bash
# JWT/HMAC secret 하드코딩 여부
grep -r "jwt.secret\|JWT_SECRET\|hmac.*secret" --include="*.yml" --include="*.java" .

# Redis 비밀번호 하드코딩 여부
grep -r "redis.*password" --include="*.yml" .

# DB 비밀번호 확인
grep -r "spring.datasource.password" --include="*.yml" .
```

**목표**: 모든 시크릿이 `${ENV_VAR:defaultForDev}` 형태이고 `.env`에 정의됨.

## Session 4 — 인증/인가 보안 강화

**목표**: JWT 토큰 생명주기 보안, 로그아웃 토큰 무효화, 면제 경로 최소화.

### 4-A. JWT 만료 시간 및 설정 검토

**확인 항목**:
```java
// JwtVerifier 또는 JwtProvider
// accessToken expiry: 15분 ~ 1시간 권장
// refreshToken expiry: 7~30일 권장
```

**문제 없으면 통과, 과도하게 길면 조정.**

### 4-B. 로그아웃 시 refreshToken 블랙리스트 (Session 2 B-NEW-6 연동)

**현황**: `POST /api/auth/logout`이 Redis blacklist에 등록하는지 확인 필요.

**구현 (없을 경우)**:

```java
// auth-service AuthService.logout()
// 기존 코드에 추가:
String tokenKey = "blacklist:rt:" + refreshTokenHash;
redisTemplate.opsForValue().set(tokenKey, "1", refreshTokenTtl, TimeUnit.MILLISECONDS);

// auth-service TokenRefreshFilter 또는 refresh endpoint
// refresh 요청 시 blacklist 확인:
if (redisTemplate.hasKey("blacklist:rt:" + hash)) {
    throw new BusinessException(ErrorCode.TOKEN_REVOKED);
}
```

**Redis DB 인덱스**: `auth=0` 사용 (P5 규칙).
**키 패턴**: `blacklist:rt:{sha256(refreshToken)}` — ERD.md §5에 추가.

## Session 5 — 매크로 차단 시스템 구축

**목표**: CaptchaStubFilter의 실질적 강화, 로그인 브루트포스 방지, 봇 User-Agent 차단.

### 5-A. 로그인 브루트포스 방지 (신규 GlobalFilter)

**대상 경로**: `/api/auth/login`, `/api/auth/guest/login`
**로직**: IP + 경로 기준, 1분 내 5회 실패 시 5분 block

**구현 위치**: `api-gateway` 신규 `BruteForceFilter` (GlobalFilter, order -60)

```java
// 실패는 downstream(auth-service) 응답 401/403 감지
// Redis key: bruteforce:{ip}:{path}:{minuteWindow}  TTL: 5분
// 5회 초과 → 429 + Retry-After: 300

// 주의: 인증 성공 시 카운터 리셋 (선택적 최적화)
```

**Redis DB**: `auth=0` 사용.
**키 패턴**: `bruteforce:{ip}:{path_hash}:{minuteEpoch}`

### 5-B. User-Agent 기반 봇 탐지 필터

**대상**: 알려진 봇 User-Agent 패턴 차단

**구현 위치**: `api-gateway` 신규 `BotDetectionFilter` (GlobalFilter, order -45)

```java
// 차단 패턴 (예시)
private static final List<String> BOT_PATTERNS = List.of(
    "python-requests", "curl/", "Go-http-client", "Apache-HttpClient",
    "Scrapy", "mechanize", "wget/", "libwww-perl"
);

// User-Agent 없는 요청도 봇으로 간주 → 요청 횟수 제한 강화
// 단, /actuator/health는 제외 (health check 클라이언트)
```

**주의**: false positive 방지를 위해 완전 차단이 아닌 `X-Suspicious: 1` 헤더 추가 후 Rate Limit 절반으로 적용하는 방식 권장.

### 5-C. CaptchaStubFilter 강화

**현황**: `CAPTCHA_PATHS` = `/api/auth/signup`, `/api/queue/token` — 항상 통과.

**강화 방안**: 실제 CAPTCHA 연동 없이도 의미 있는 검증 추가.

**Option A — Proof-of-Work stub** (추천):
- 클라이언트가 `X-Captcha-Token: c_stub:{nonce}` 전송
- nonce가 특정 패턴 (예: 4자리 숫자) 이어야 통과
- 프론트엔드에서 nonce 생성 로직 추가

**Option B — 시간 기반 토큰**:
- `X-Captcha-Token: {base64(timestamp)}` 형태
- 토큰 발급 후 30초 이내 사용만 허용 (재사용 방지)
- 서버에서 timestamp 디코딩 후 유효성 검증

**최종 결정**: Option B 구현 (프론트 변경 최소화, 재사용 방지 효과).

**구현**:
```java
// CaptchaStubFilter 수정
// X-Captcha-Token 형식: base64(timestamp_ms)
// 유효 조건: |now - timestamp| <= 30_000ms AND 해당 토큰 Redis에 없음 (일회성)
// Redis key: captcha:used:{token}  TTL: 60s
```

### 5-D. 대기열 토큰 재사용 방지 강화

**현황**: `QueueTokenInterceptor` — HMAC 서명 검증. scope=global.

**추가 확인**:
- `queueToken`이 일회성인지 (사용 후 Redis에서 삭제되는지) 검증
- `/api/reservations` POST 이후 토큰 무효화 여부 확인
- 토큰 유효 시간(TTL) 확인 — 너무 길면 재사용 위험

## Session 6 — 비정상 접근 방어체계 통합 검증

**목표**: 구현된 모든 방어 레이어를 종합 테스트하고 최종 보안 상태 문서화.

### 6-A. Rate Limit 세분화

**현황**: 전역 100/min, `/api/reservations` 30/min.

**추가 엔드포인트별 설정**:

| 경로 | 현재 | 변경 | 이유 |
|---|---|---|---|
| `/api/auth/login` | 100/min | 10/min per IP | 브루트포스 방지 (Session 5-A와 이중 방어) |
| `/api/auth/signup` | 100/min | 5/min per IP | 계정 대량 생성 방지 |
| `/api/payments` | 100/min | 5/min per IP | 결제 반복 시도 방지 |
| `/api/queue/token` | 100/min | 20/min per IP | 대기열 진입 매크로 방지 |

**구현**: `RateLimitFilter`의 capacity 분기 로직 확장.

### 6-B. 통합 공격 시나리오 테스트

**시나리오 1: 브루트포스 로그인**
```bash
for i in $(seq 1 10); do
  curl -X POST http://localhost:8080/api/auth/login \
    -H 'Content-Type: application/json' \
    -d '{"email":"test@test.com","password":"wrong"}' &
done
# 기대: 5회 이후 429 + Retry-After 반환
```

**시나리오 2: 헤더 스푸핑 시도**
```bash
curl http://localhost:8080/api/reservations \
  -H 'X-User-Id: 1' \
  -H 'X-User-Role: ADMIN'
# 기대: 401 (Authorization 헤더 없어서 JWT 검증 실패, X-User-* 무시됨)
```

**시나리오 3: 봇 User-Agent 차단**
```bash
curl http://localhost:8080/api/trains \
  -H 'User-Agent: python-requests/2.28.0'
# 기대: 429 또는 X-Suspicious 헤더 포함 응답
```

**시나리오 4: CAPTCHA 토큰 재사용**
```bash
TOKEN=$(echo -n "1748870400000" | base64)
# 동일 토큰 2회 사용 → 두 번째 요청 실패
curl -X POST http://localhost:8080/api/auth/signup \
  -H "X-Captcha-Token: $TOKEN" ...
curl -X POST http://localhost:8080/api/auth/signup \
  -H "X-Captcha-Token: $TOKEN" ...  # 기대: 403
```

**시나리오 5: 이중 결제 방지**
```bash
KEY=$(uuidgen)
curl -X POST http://localhost:8080/api/payments \
  -H "Idempotency-Key: $KEY" -d '{"reservationId":1,"method":"CARD"}' &
curl -X POST http://localhost:8080/api/payments \
  -H "Idempotency-Key: $KEY" -d '{"reservationId":1,"method":"CARD"}' &
```

**시나리오 6: 로그아웃 후 refreshToken 재사용**
```bash
REFRESH=$(# localStorage에서 복사)
curl -X POST http://localhost:8080/api/auth/logout -H "Authorization: Bearer <token>"
curl -X POST http://localhost:8080/api/auth/refresh \
  -H 'Content-Type: application/json' \
  -d "{\"refreshToken\":\"$REFRESH\"}"
# 기대: 401 TOKEN_REVOKED
```
