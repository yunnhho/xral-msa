## S5~S8 세션 테스트 중 발견된 버그 {#s5-s8}

### B1. QueueTokenInterceptor scope 기본값 불일치 — CRITICAL

**증상**: POST `/api/reservations` 호출 시 401 Unauthorized 반환. 모든 예약 불가.

**원인**:
- 소스 코드: `QueueTokenInterceptor.java` → `scope = "global"` (수정됨)
- 실행 중인 빌드: `scope = "reservation"` (구 버전 컴파일 클래스)
- 큐 서비스가 "global" scope로 HMAC 토큰 발급 → train-service가 "reservation"으로 검증 → HMAC 불일치 → 401

**수정**:
```bash
./gradlew :train-service:compileJava  # 재컴파일
# 서비스 재시작
```

**영향 파일**: `train-service/src/main/java/com/xrail/train/interceptor/QueueTokenInterceptor.java`

### B2. SagaLog 락 타임아웃으로 게이트웨이 타임아웃 — HIGH

**증상**: 예약 시 `TRAIN_SERVICE_UNAVAILABLE` 반환. 실제로는 예약이 DB에 저장되나 응답이 오지 않음.

**원인**:
- `expireReservation()` 내부: `reservation.cancel()` 이 `reservations` 테이블 행에 exclusive lock 획득
- 이후 `sagaLogService.recordOutbound()` (REQUIRES_NEW) 가 동일 행 FK 체크를 위해 shared lock 요청
- MySQL innodb_lock_wait_timeout(50초) 대기 → 게이트웨이 5초 timelimiter 먼저 발화

**수정**:
1. `SagaLogService` - timeout=3초 추가
```java
@Transactional(propagation = Propagation.REQUIRES_NEW, timeout = 3)
public void recordOutbound(...) { ... }
```
2. `ReservationService` - `handlePaymentCompleted`, `expireReservation`, `handlePaymentFailed` 에 saga log 호출부 try/catch 추가 (이미 doCreate는 있었음)
```java
try {
    sagaLogService.recordInbound(reservationId, Topics.PAYMENT_COMPLETED, reservationId);
} catch (Exception e) {
    log.warn("Saga log write failed reservationId={} reason={}", reservationId, e.getMessage());
}
```

**영향 파일**:
- `train-service/.../service/SagaLogService.java`
- `train-service/.../service/ReservationService.java`

### B3. ReservationResponse DTO — tickets 필드 누락 — HIGH

**증상**: PaymentPage 진입 시 React 오류 `TypeError: Cannot read properties of undefined (reading 'map')`.

**원인**: 백엔드 `ReservationResponse`가 `ticketIds: List<Long>` 반환, 프론트는 `tickets[].seatNumber` 기대.

**추가 누락 항목**:
- `reservedAt` 필드 미포함
- `seatNumber` 필드 미포함 (Ticket 엔티티는 seatId만 저장)

**수정**:
```java
// ReservationResponse.java
public record ReservationResponse(
    Long reservationId, Long userId, String status, Long totalPrice,
    LocalDateTime reservedAt, LocalDateTime expiresAt,
    List<TicketSummary> tickets  // ← 추가
) {
    public record TicketSummary(Long ticketId, Long seatId, String seatNumber, Long price) {}
    
    public static ReservationResponse of(Reservation r, List<Ticket> tickets, Map<Long, String> seatNumberMap) {
        // seatNumberMap: SeatRepository.findAllById로 조회
    }
}
```
- `ReservationService.toResponse()` 헬퍼 추가 (SeatRepository 주입)
- `IdempotencyInterceptor` 도 같은 방식으로 수정

**영향 파일**:
- `train-service/.../dto/ReservationResponse.java`
- `train-service/.../service/ReservationService.java`
- `train-service/.../interceptor/IdempotencyInterceptor.java`

### B4. Kafka Consumer 무한 재처리 (handlePaymentCompleted) — HIGH

**증상**: `payment.completed` 이벤트 수신 후 `reservations.status` 가 PAID로 업데이트 안 됨. 로그에 "Seeking to offset 0" 반복.

**원인**:
- `handlePaymentCompleted()` 내 `sagaLogService.recordInbound()` 호출에 try/catch 없음
- saga log 타임아웃 예외 발생 → `@Transactional` 롤백 → Kafka consumer 오프셋 미커밋 → 무한 재처리

**수정**: B2와 동일하게 try/catch 추가 (이미 B2 수정으로 해결됨)

### B6. 결제 후 예매내역에서 PENDING으로 표시 — MEDIUM

**증상**: 결제 성공 후 `/reservations` 이동 시 방금 결제한 예약이 "결제 대기"로 표시됨.

**원인**: Kafka `payment.completed` → train-service 처리 지연 (2~5초). 페이지 즉시 로드 시 DB 업데이트 전.

**수정** (`ReservationsPage.tsx`):
```typescript
// justPaid + reservationId 수신 시 2초 간격 폴링
useEffect(() => {
    if (!justPaid || !paidId) return
    let attempts = 0
    pollRef.current = setInterval(() => {
        attempts++
        client.get('/api/reservations').then(({ data }) => {
            const list = data.data ?? []
            setReservations(list)
            const paid = list.find(r => r.reservationId === paidId && r.status === 'PAID')
            if (paid || attempts >= 10) {
                clearInterval(pollRef.current!)
                if (paid) setPaidConfirmed(true)
            }
        })
    }, 2000)
}, [justPaid, paidId])
```
- 폴링 완료 시 "결제가 완료되었습니다." 배너 표시

### Bug #2 — 서비스 재시작 후 점유 좌석 선택 가능 + 예약 불가

**증상**: 좌석이 선택 가능(파랑)으로 표시되나 예약 클릭 시 `SEAT_ALREADY_TAKEN` 반환.

**원인**: 서비스 재시작 시 Redis bitmask 초기화 → `isFree()=true` → 좌석이 available로 표시. 그러나 DB에는 RESERVED 상태 티켓 존재 → DB 이중 체크에서 걸림.

**3중 수정**:

**① 시작 시 DB→Redis 복원** (`ReconciliationScheduler.java`):
```java
@EventListener(ApplicationReadyEvent.class)
public void restoreOnStartup() {
    List<Ticket> active = ticketRepository.findByStatusIn(List.of(RESERVED, ISSUED));
    for (Ticket t : active) {
        luaScriptService.tryReserve(t.getScheduleId(), t.getSeatId(), ...);
    }
}
```

**② 5분 주기 양방향 동기화** (기존 단방향 → 양방향):
- 기존: phantom 비트 제거 (Redis에만 있는 비트 삭제)
- 추가: 누락 비트 복원 (DB에 있지만 Redis에 없는 비트 재설정)

**③ DB 충돌 시 즉시 복원** (`ReservationService.java`):
```java
if (dbConflict) {
    luaScriptService.tryReserve(scheduleId, seatId, startIdx, endIdx); // bitmask 재설정
    rollbackLockedSeats(lockedSeats, ...); // 이 seatId는 lockedSeats에 없어서 유지됨
    throw new BusinessException(ErrorCode.SEAT_ALREADY_TAKEN);
}
```

**검증**:
```
STEP1: Redis DEL "sch:2:seat:1" → 1A available: True  (버그 재현)
STEP2: 예약 시도 → SEAT_ALREADY_TAKEN  (DB 체크 정상)
STEP3: 이후 좌석 조회 → 1A available: False  (자동 복원 ✓)
```

### Bug #4 — 취소된 예약 좌석이 여전히 점유됨 (예약 불가)

**증상**: 예매내역에서 예약 취소 후, 해당 좌석을 다시 선택하면 `SEAT_ALREADY_TAKEN` 반환.

**원인**: `TicketRepository.existsBy...` 메서드에 **status 필터 없음** → CANCELLED 티켓도 충돌로 인식.

```java
// Before (모든 상태 포함)
boolean existsByScheduleIdAndSeatIdAndStartStationIdxLessThanAndEndStationIdxGreaterThan(...)

// After (CANCELLED 제외)
boolean existsByScheduleIdAndSeatIdAndStartStationIdxLessThanAndEndStationIdxGreaterThanAndStatusIn(
    Long scheduleId, Long seatId, int endIdx, int startIdx,
    Collection<TicketStatus> statuses  // RESERVED, ISSUED 만
)
```

**수정** (`ReservationService.java`):
```java
boolean dbConflict = ticketRepository
    .existsByScheduleIdAndSeatIdAndStartStationIdxLessThanAndEndStationIdxGreaterThanAndStatusIn(
        request.scheduleId(), seatId, endIdx, startIdx,
        List.of(TicketStatus.RESERVED, TicketStatus.ISSUED));  // CANCELLED 제외
```

**검증**:
- CANCELLED 좌석 (1A, 1B, 2A, 10A) → 다시 선택 가능 ✓
- PENDING 좌석 → 여전히 막힘 ✓
- PAID 좌석 → 여전히 막힘 ✓

## 서비스 운영 이슈 {#운영-이슈}

### API Gateway 라우팅 추가

```yaml
# application.yml — payment-service 라우트
predicates:
  - Path=/api/payments/**, /api/coupons/**  # /api/coupons 추가
```

### 게이트웨이 서킷브레이커 반복 개방 원인

서비스 재시작 직후 Circuit Breaker 가 HALF_OPEN → OPEN 반복. 원인:
- 재시작 직후 DB/Redis 연결 완료 전 헬스체크 통과
- 첫 몇 개 요청이 느려 timelimiter(5초) 발화

> 대응: 재시작 후 15초 대기 후 요청 진행.

### train-service 빌드 클래스 불일치 주의

`./gradlew :train-service:bootRun` 은 `build/classes/java/main/` 디렉토리를 직접 사용.
소스 변경 후 `compileJava` 또는 `bootRun` 실행 필수. 이전 프로세스 kill 후 재시작 필요.

### SagaLog DB 락 근본 원인

`expireReservation()` 에서 `reservation.cancel()` 실행 → reservations 행 exclusive lock 획득 상태에서
`sagaLogService.recordOutbound()` (REQUIRES_NEW) 가 동일 행 FK 검증을 위해 shared lock 요청 → 교착.

> 근본 해결책: Transactional Outbox 패턴 적용 권고 (현재 deferred).
