## 1. 좌석 동시성

### 1-1. 동시 요청에서의 좌석 중복 판매

- **문제(예상)**: 티켓 오픈 직후 수천~수만 동시 요청이 같은 좌석에 몰린다. DB `SELECT → INSERT` 사이의 시간차(check-then-act)로 동일 좌석·동일 구간이 중복 판매될 수 있다. 비관적 락은 락 경합으로 처리량이 무너지고, 단순 유니크 제약은 "구간 겹침"(서울→대전 vs 대전→부산은 공존 가능)을 표현하지 못한다.
- **해결**: Redis **Lua 스크립트 + 비트마스크**. 좌석×스케줄당 키(`sch:{id}:seat:{id}`)에 역 구간을 비트(`[startIdx, endIdx-1]`)로 표현하고, `reserve_seat.lua`가 "겹침 검사 + 비트 셋"을 **단일 원자 연산**으로 수행. 부분 실패 시 이미 잡은 비트는 `rollback_seat.lua`로 즉시 해제.
- **결과**: 구간 겹침 판정이 Redis 단일 호출(O(1))로 끝나 DB 락 경합 없이 동시성 제어. 겹치지 않는 구간은 같은 좌석이라도 동시 판매 가능(좌석 활용률↑). JMeter 동시 부하에서 중복 예약 0건.

### 1-2. Redis만 믿을 수 없다 — 2중 방어선

- **문제(예상)**: Redis 장애·플러시·복구 시점에 비트마스크가 유실되면 Lua 검사를 통과한 요청이 실제로는 팔린 좌석을 다시 판매할 수 있다.
- **해결**: 예약 트랜잭션 안에서 **DB 겹침 더블체크**(`tickets` 테이블 구간 overlap 쿼리, RESERVED/ISSUED만 대상)를 Lua 잠금과 병행. DB 충돌 발견 시 누락된 Redis 비트를 즉시 복원(자가 치유)하고 `SEAT_ALREADY_TAKEN` 반환.
- **결과**: Redis가 비어 있어도 중복 판매가 DB 단계에서 차단되고, 차단과 동시에 Redis 상태가 복구된다. "DB가 진실(source of truth), Redis는 성능 캐시"라는 원칙 확립.

## 2. 대기열

### 2-1. 오픈런 트래픽이 예약 시스템을 직격

- **문제(예상)**: 명절 승차권 오픈 같은 순간 폭주 트래픽이 예약/결제 API에 그대로 꽂히면 DB 커넥션 고갈로 전체 장애가 난다.
- **해결**: 코레일·Queue-it 방식의 **가상 대기열**. Redis Sorted Set(score=진입 시각)으로 순번을 관리하고, 스케줄러가 3초마다 100명씩 승급. 승급자에게 **HMAC 서명 큐 토큰**(`HMAC(userId:scope:issuedAt)`)을 발급하고, train-service 인터셉터가 서명·만료·**일회성 소비**(Redis used-key)를 검증해 토큰 없는 직접 호출을 차단. 대기 순번은 SSE로 실시간 푸시(에러 2회 시 polling fallback).
- **결과**: 예약 API 유입이 초당 ~33명으로 평탄화되어 백엔드가 받는 부하 상한이 고정된다. 토큰 위조·재사용·대기열 우회가 모두 차단되고, 사용자는 실시간 순번/예상 대기시간을 확인.

### 2-2. 평시에도 항상 줄을 세우는 UX 낭비

- **문제**: 초기 구현은 부하와 무관하게 무조건 대기열에 등록 → 한산한 시간에도 스케줄러 주기(최대 3초)를 기다려야 했다.
- **해결**: **하이브리드 입장 제어**. `queue:mode`(AUTO/FORCE_ON/FORCE_OFF, Redis 저장)와 동시 active 임계치(`max-active`, active ZSet으로 카운팅)를 도입 — AUTO는 "대기자 0 + active 여유"면 즉시 토큰 발급, 초과 시 자동 대기열. 운영자는 admin API/콘솔에서 모드를 실시간 전환(명절엔 FORCE_ON).
- **결과**: 평시 0초 통과 + 폭주 시 자동 대기열 활성화 + 운영자 수동 override까지, 상용 대기열 솔루션과 같은 3단 운영이 가능.

### 2-3. SSE의 수평 확장 한계

- **문제(예상)**: SSE emitter는 인스턴스 메모리에 있다. 승급 스케줄러가 다른 인스턴스에서 돌면 해당 사용자의 SSE로 알림을 보낼 수 없다.
- **해결**: 승급 알림을 emitter 직접 호출 대신 **Redis pub/sub(RTopic)**으로 발행하고, 각 인스턴스가 구독해 자기 emitter에만 전달. 구독자 0이면 로컬 emitter로 fallback.
- **결과**: queue-service를 N대로 늘려도 어느 인스턴스에 SSE가 붙어 있든 알림이 도달한다.

## 3. 분산 트랜잭션

### 3-1. 서비스별 DB 분리로 인한 정합성 문제

- **문제(예상)**: Database-per-service 원칙상 "좌석 확정 + 결제 성공"을 하나의 DB 트랜잭션으로 묶을 수 없다. 결제는 성공했는데 좌석 확정이 실패하면(또는 그 반대) 돈만 나가거나 좌석만 잠긴다.
- **해결**: **Saga Choreography**(오케스트레이터 없음). `reservation.created → payment.completed/failed → seat.confirmed/released`의 이벤트 체인으로 진행하고, 실패 시 보상(좌석 비트 롤백 + 예약 CANCELLED + `seat.released` 발행)은 **train-service가 단일 책임**. PAID 취소는 역방향 환불 사가(`payment.refund-requested → payment.refunded`)로 처리.
- **결과**: 결제 실패·타임아웃·운영자 강제취소·환불 등 5개 실패 시나리오가 자동 복구된다. E2E에서 MockPG 10% 랜덤 실패 유발 → 좌석 자동 반환·예약 자동 취소 확인.

### 3-2. "DB 커밋은 됐는데 이벤트가 유실"되는 이중 쓰기 문제

- **문제(예상)**: 트랜잭션 안에서 `kafkaTemplate.send()`를 직접 호출하면 ① 커밋 전에 이벤트가 먼저 나가 롤백 시 유령 이벤트가 퍼지고, ② 커밋 후 브로커 장애면 이벤트가 영영 유실된다. 사가 체인은 이벤트 하나만 빠져도 멈춘다.
- **해결**: **Transactional Outbox 패턴**. 이벤트를 비즈니스 트랜잭션과 **같은 트랜잭션에서 `outbox_events` 테이블에 INSERT**하고, relay 스케줄러(1초 폴링)가 커밋된 PENDING 행을 Kafka로 발행 후 SENT 마킹. payload를 원래 이벤트 타입으로 역직렬화해 발행하므로 `__TypeId__` 헤더가 직접 발행과 동일(컨슈머 무수정). payment-service에 먼저 적용 후 사가의 최종 책임자인 train-service까지 확장.
- **결과**: DB 커밋과 이벤트 발행의 원자성 보장(유실 0). at-least-once 발행은 컨슈머 멱등으로 흡수. admin 콘솔에서 양 서비스 outbox 적체(pending/최장 미발행 경과초)를 실시간 감시 — E2E 기준 전 토픽 SENT·대기 0.

### 3-3. 컨슈머 실패 → 무한 재시도 루프

- **문제**: 컨슈머가 예외를 그냥 던지면 같은 메시지를 무한 재소비한다. 파티션이 막혀 뒤의 정상 메시지까지 처리 불가(초기 train-service에 에러 핸들러 누락으로 실제 존재했던 위험).
- **해결**: 전 서비스 `DefaultErrorHandler + DeadLetterPublishingRecoverer`(1초 백오프 × 2회 재시도 후 `.DLT` 토픽 격리). DLT 컨슈머는 `DltAlertLog` 테이블에 저장하고 admin API/콘솔로 노출. DLT 핸들러 자신은 절대 throw하지 않음(2차 루프 방지).
- **결과**: poison message가 본 파이프라인을 막지 못하고, 격리된 메시지는 운영자가 콘솔에서 확인 후 수동 조치 가능.

### 3-4. 중복·역순 이벤트로 인한 상태 오염

- **문제**: at-least-once 환경에서는 같은 이벤트가 2번 오거나 순서가 뒤집힐 수 있다. 실제로 중복 `payment.failed`가 보상 로직을 재실행해 **다른 예약의 좌석 비트를 풀어버리는** 동시성 버그가 있었다.
- **해결**: 모든 이벤트에 `eventId(UUID)` 포함 + 컨슈머 멱등 처리. 상태 전이에는 가드를 명시 — `handlePaymentFailed`는 `status == PENDING`일 때만 보상, `handlePaymentCompleted`는 이미 PAID면 no-op, 환불은 COMPLETED 결제가 없으면 no-op.
- **결과**: 중복/역순 이벤트가 와도 상태 기계가 한 방향으로만 전이. 같은 reservationId는 같은 Kafka 파티션(키 지정)이라 순차 처리도 보장.

## 4. 멱등성

### 4-1. 중복 결제(이중 청구)

- **문제(예상)**: 네트워크 타임아웃 후 클라이언트 재시도, 더블 클릭 등으로 같은 결제가 2번 실행되면 이중 청구 사고가 난다.
- **해결**: 3중 장치 — ① `Idempotency-Key`별 **Redisson 분산락**(instant fail, 충돌 시 409), ② Redis 멱등 버킷(`payment:idem:{key}` PROCESSING→COMPLETED, 완료 키 재요청은 기존 응답 반환), ③ DB unique + `@Version` 낙관적 락. 락은 `TransactionTemplate`으로 **트랜잭션 바깥**에서 잡아 "락 해제 후 커밋 전" 틈을 제거.
- **결과**: 같은 키 재요청은 몇 번을 보내도 청구 1회 + 동일 응답. 처리 중 재진입은 409로 명시 거부.

### 4-2. 중복 예약 생성

- **문제(예상)**: 예약 POST 재시도 시 같은 좌석이 두 예약으로 잡힐 수 있다.
- **해결**: `IdempotencyInterceptor`(Redis SETNX 5분) + `reservations.idempotency_key` 조회로 기존 예약 재반환. 큐 토큰 일회성 소비(§2-1)가 추가 방어선.
- **결과**: 인터셉터→토큰→Lua→DB의 4중 그물로 동일 요청 중복 생성 0건.

## 5. 게이트웨이 보안

### 5-1. 마이크로서비스마다 인증을 구현하는 비용과 스푸핑 위험

- **문제(예상)**: 6개 서비스가 각자 JWT를 검증하면 시크릿 배포·로직 불일치 위험이 있고, 내부 신뢰 헤더(`X-User-*`)를 클라이언트가 직접 보내 권한을 위조할 수 있다.
- **해결**: **Gateway 단일 인증**. `JwtValidationFilter` 한 곳에서만 검증하고, `HeaderStripFilter`가 inbound `X-User-Id/Role/Name`을 **무조건 제거 후** 파싱 값으로 재주입(제거가 주입보다 항상 먼저). downstream은 헤더를 신뢰만 하고 자체 JWT 로직 금지.
- **결과**: 인증 로직 단일화 + 헤더 스푸핑 원천 차단. admin 권한도 같은 경로로 일원화(가드가 `"ADMIN"` 리터럴을 비교해 실제 `ROLE_ADMIN` 토큰이 거부되던 버그를 `Roles.ADMIN` 상수로 교정).

### 5-2. 매크로·봇의 좌석 선점

- **문제(예상)**: 자동화 스크립트가 대기열·예약을 선점하면 일반 사용자가 피해를 본다.
- **해결**: 다층 방어 — IP Rate Limit(**Redis Lua 원자 카운터**, 인스턴스 간 공유, Redis 장애 시 로컬 fallback + `Retry-After` 헤더), 시간 기반 CAPTCHA 토큰(서명+재사용 차단, `/signup`·`/queue/token`에 적용), 로그인 브루트포스 카운터, 봇 시그니처 탐지, 응답 보안 헤더.
- **결과**: 게이트웨이 1차 방어 + 큐 토큰/멱등키 2차 방어의 직렬 구조. 분산 환경에서도 rate limit 카운터가 일관(과거 인메모리 Map 구현은 인스턴스별 N배 허용이라 Redis Lua로 교체).

## 6. 정합성 복구

### 6-1. 미결제 예약의 좌석 영구 점유

- **문제(예상)**: 좌석만 잡고 결제하지 않은 예약이 남으면 그 좌석은 영영 팔 수 없다.
- **해결**: 예약에 20분 만료(`expires_at`)를 두고 `ReservationExpiryScheduler`(60초)가 만료 PENDING을 건별 보상(Lua 롤백 + CANCELLED + `seat.released`). 트랜잭션 안에서 **재조회 후 상태 재확인** — detached 엔티티는 dirty-check가 안 돼 만료 처리가 매번 롤백되며 무한 반복되던 버그(F-14)와, 스케줄러 조회~처리 사이에 결제가 완료된 예약을 취소하는 race를 함께 차단.
- **결과**: 미결제 좌석이 최대 21분 내 자동 회수. PAID 전환된 예약은 절대 만료 취소되지 않음.

### 6-2. Redis 비트와 DB의 장기 드리프트

- **문제(예상)**: 보상 누락·장애로 "DB엔 예약이 없는데 Redis 비트만 켜진" 유령 좌석이 누적되면 판매 가능 좌석이 줄어든다.
- **해결**: `ReconciliationScheduler`(5분)가 Redis 비트 ↔ DB RESERVED/PAID를 대조해 고아 비트만 해제. 단, **연속 2회 관측된 경우에만** 해제(2-스트라이크) — 진행 중 예약(Lua 선점 후 DB INSERT 전)을 오삭제하던 race를 방지.
- **결과**: 어떤 장애 경로로 정합성이 깨져도 최대 10분 내 자동 수렴. in-flight 예약 오삭제 0건.

## 7. 관측성 & 운영

### 7-1. MSA에서 요청 흐름·장애 지점 추적 불가

- **문제(예상)**: 하나의 예매가 4개 서비스 + Kafka를 거치면 "어디서 끊겼는지"를 로그만으로 찾기 어렵다.
- **해결**: Micrometer Tracing + Zipkin(traceId 전파, Kafka 구간 포함), 전 서비스 로그 패턴에 `traceId/spanId/userId` MDC, 도메인 메트릭(`xrail.*`) Prometheus 수집 + Grafana 대시보드. 사가 흐름은 `reservation_saga_log`(OUTBOUND/INBOUND)로 영속 기록.
- **결과**: 단일 traceId로 게이트웨이→예약→결제→알림 전 구간 추적. 대기열 크기·승급 수·결제 성공/실패가 시계열로 가시화.

### 7-2. 운영자가 시스템 상태를 보고 개입할 수단 부재

- **문제**: 통계·강제 조치가 전부 DB 직접 쿼리에 의존했다.
- **해결**: 운영자 콘솔(React `/admin`, ROLE_ADMIN 가드) — 예약/결제/알림 통계, **양 서비스 Outbox 적체 감시**, 사가 로그 뷰어, DLT 알림 통합 조회, 예약 강제취소(환불 사가 연동), 대기열 모드 제어.
- **결과**: 조회→진단→조치(강제취소·모드 전환)가 콘솔에서 완결. E2E로 강제취소→환불 사가 전 구간 검증.

## 8. E2E 결함 사례

> 단위 테스트 122개가 전부 통과하는 상태에서, 실구동 + Playwright E2E로만 잡힌 결함들.
> "테스트가 초록색"과 "시스템이 정상"은 다르다는 것을 보여준 사례.

### 8-1. 사가 로그가 한 번도 기록된 적이 없었다 — FK 자기-교착

- **문제**: `reservation.created`/`payment.completed` 사가 로그가 **프로젝트 전 기간 동안 단 1건도 DB에 없었다**(auto-increment 갭이 증거). 단위 테스트는 mock이라 통과했고, 실패는 try-catch로 삼켜져 조용히 유실됐다.
- **해결(원인 규명)**: `reservation_saga_log`의 reservations **FK**가 범인. REQUIRES_NEW로 도는 로그 INSERT가 FK 검증을 위해 부모 행의 S-lock을 기다리는데, 그 행의 X-lock은 외부 비즈니스 트랜잭션이 쥔 채 로그 메서드의 리턴을 기다림 → **자기-교착** → `timeout=3` statement timeout. 외부 트랜잭션이 해당 행을 이미 flush한 경로(생성=IDENTITY 즉시 flush, 결제완료=markPaid 후 SELECT의 auto-flush)만 실패하고, 취소 경로는 flush 전에 로그를 써서 성공 — 그래서 "일부만 기록"되는 형태로 은폐됐다. **V6 마이그레이션으로 FK 제거**(디버깅 로그 테이블에 참조 무결성 강제는 불필요) + 경고 로그에 스택트레이스 추가.
- **결과**: created/completed/failed 사가 로그 최초 기록 확인. 교훈: ① REQUIRES_NEW 트랜잭션과 부모 행 FK는 자기-교착 조합, ② 삼키는 catch에는 반드시 스택트레이스, ③ 로그성 테이블에 FK를 걸지 않는다.

### 8-2. 두 번째 예매가 무조건 실패 — 소비된 큐 토큰 재사용

- **문제**: 예매 성공 직후 10분 안에 다시 예매하면 401. 일회성 토큰은 예약 POST에서 소비되는데, queue의 active 버킷은 TTL(600초)까지 **그 소비된 토큰을 계속 재반환**했다. 두 서비스가 토큰 수명을 다르게 이해한 정합성 버그.
- **해결**: ① 예약 성공 시 프론트가 `POST /api/queue/leave`(fire-and-forget) — 소비된 토큰 버킷 제거 + **active 슬롯 즉시 반납**(입장 제어 정확도 개선 부수 효과), ② 예약 POST 401 시 leave 후 대기열 자동 재진입(자기복구 경로).
- **결과**: Playwright로 결정적 재현 → 수정 → 연속 예매 무중단 + 스테일 상태 자기복구까지 검증. 대기열 active 카운트도 실사용과 일치.

### 8-3. 예약 시각이 9시간 미래로 표시

- **문제**: 13:20에 예약했는데 "오후 10:20"으로 표시. `LocalDateTime`(JVM 로컬 벽시계)을 `.atOffset(UTC)`로 **거짓 태깅** → KST 브라우저가 +9h 변환. 결제 만료 타이머도 동일하게 틀어짐. (docker(UTC) 환경에서는 우연히 맞아 발견이 늦었다.)
- **해결**: `.atZone(ZoneId.systemDefault())`로 실제 시스템 존을 태깅(ReservationResponse/PaymentResponse/TrainEventProducer). UTC 컨테이너에서는 기존과 동일하게 동작.
- **결과**: 로컬(KST)·docker(UTC) 양쪽에서 시각 표시 정상. 교훈: 시간은 저장-직렬화-표시 전 구간에서 존을 명시해야 하며, "벽시계에 UTC 라벨 붙이기"는 환경 의존 버그를 만든다.

### 8-4. 잘못된 쿠폰 코드로 전액 결제

- **문제**: 결제 API에 존재하지 않는 쿠폰 코드를 보내면 **조용히 무시되고 전액 청구**되면서 Payment 레코드에는 couponCode가 남았다(할인 기대한 사용자에게 전액 청구 + 오해를 부르는 데이터). 검증 엔드포인트는 같은 코드를 거부하는 비일관.
- **해결**: 결제 경로의 `applyDiscount`도 `COUPON_INVALID`로 거부 — Payment INSERT·PG 호출 전에 차단. 단위 테스트로 거부 시 부수효과 없음을 고정.
- **결과**: validate/pay 두 경로의 쿠폰 의미가 일치. 우회 호출로도 의도치 않은 전액 청구 불가.

### 8-5. 자잘한 결함 일괄 수정

- **문제**: ① React StrictMode 이중 마운트로 대기열 enter가 중복 POST되어 CAPTCHA 재사용 차단(403)에 걸림, ② 대기열 waiting 게이지가 scope 태그 없이 등록되어 전 scope가 한 게이지로 합쳐짐, ③ 결제 직후 예매내역의 "결제 확인 중" 배너가 첫 폴링 주기(2초)까지 남음, ④ favicon 404 콘솔 노이즈.
- **해결**: ① effect ref 가드, ② `Tags.of("scope", ...)` 추가, ③ 폴링 시작 전 즉시 1회 확인, ④ 인라인 SVG favicon.
- **결과**: 콘솔 에러 0, 메트릭 scope별 분리, 결제 직후 UX 매끄러움.
