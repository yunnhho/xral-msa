# 2026-06-03 버그 추출 목록

오늘 세션에서 발견·수정된 버그 및 코드베이스 전체를 스캔하여 추출한 잠재적 결함 목록.

## ✅ 수정 완료

### F-01 · `discovery-server` healthcheck — `curl` 미설치
- **위치**: `docker-compose.yml` → `discovery-server.healthcheck`
- **원인**: Alpine JRE 이미지에 `curl` 없음 → healthcheck 실패 → downstream 서비스 전체 기동 불가
- **수정**: `curl -f` → `wget -qO-`

### F-02 · `.env` `EUREKA_DEFAULT_ZONE` localhost
- **위치**: `.env` L30
- **원인**: `http://localhost:8761/eureka/` → Docker 컨테이너 내부에서 자기 자신을 가리킴 → 모든 앱 서비스 Eureka 등록 실패 → gateway `lb://` resolve 불가 → OAuth 포함 모든 API 503
- **수정**: `http://discovery-server:8761/eureka/`

### F-03 · `ReservationResponse` `LocalDateTime` timezone 불일치
- **위치**: `train-service/.../dto/ReservationResponse.java`
- **원인**: `reservedAt`, `expiresAt` 필드가 `LocalDateTime`(시간대 없음)으로 직렬화 → 서버 UTC, 브라우저 KST(UTC+9) → `new Date("2026-06-02T16:39:02")` → KST로 파싱 → 9시간 앞당겨짐 → `isExpired()` 즉시 `true` 반환
- **증상**: 예매 내역 빈 화면, 결제 페이지 타이머 즉시 00:00
- **수정**: `LocalDateTime` → `OffsetDateTime(UTC)` (`.atOffset(ZoneOffset.UTC)`)

## 🔴 미수정 — High

### F-04 · `PaymentResponse.completedAt` timezone 불일치
- **위치**: `payment-service/.../dto/PaymentResponse.java` → `of()` 메서드
- **코드**: `p.getUpdatedAt().toString()` → `LocalDateTime.toString()` (시간대 없음)
- **증상**: 결제 완료 시각이 화면에 9시간 빠르게 표시됨
- **수정 방향**: `p.getUpdatedAt().atOffset(ZoneOffset.UTC).toString()` 또는 `Instant` 변환

### F-05 · `ReservationCreatedEvent.expiresAt` timezone 불일치
- **위치**: `train-service/.../kafka/TrainEventProducer.java`
- **코드**: `reservation.getExpiresAt().toString()` → `LocalDateTime.toString()` → 이벤트에 TZ 없는 문자열 포함
- **증상**: notification-service가 이 `expiresAt` 문자열을 알림 payload에 그대로 사용 → 사용자에게 만료 시각 9시간 오차로 전달
- **수정 방향**: `reservation.getExpiresAt().atOffset(ZoneOffset.UTC).toString()` 또는 `Instant.now()` 패턴 통일

### F-06 · `OAuth2SuccessHandler` redirect URI 하드코딩
- **위치**: `auth-service/.../oauth2/OAuth2SuccessHandler.java` L18
- **코드**: `private static final String REDIRECT_URI = "http://localhost:5173/oauth/callback";`
- **원인**: `application.yml`에 `oauth.success-redirect-uri: ${OAUTH_SUCCESS_REDIRECT:...}` 속성이 있음에도 `@Value` 주입 없이 상수로 고정
- **증상**: 배포 환경 변경 시 소스 수정 + 재빌드 필요, 환경변수로 제어 불가
- **수정 방향**: `@Value("${oauth.success-redirect-uri}") private String redirectUri;` 로 교체

### F-07 · `PaymentEventConsumer` — consumer 레벨 멱등 처리 없음
- **위치**: `train-service/.../kafka/PaymentEventConsumer.java`
- **원인**: `onPaymentCompleted` / `onPaymentFailed` 에서 `eventId` 기반 중복 처리 검사 없음. `handlePaymentCompleted`가 DB status 체크로 사실상 멱등하지만 P4 규칙("컨슈머는 반드시 멱등 처리: 이미 처리된 `eventId`이면 no-op 후 정상 커밋") 위반
- **위험**: Kafka retry/rebalance 발생 시 downstream 로직(`Ticket::issue`, Kafka emit) 재진입 가능
- **수정 방향**: Redis `SET NX` 또는 처리된 `eventId` DB 기록으로 consumer 입구에서 조기 반환

## 🟡 미수정 — Medium

### F-08 · `QueueTokenInterceptor` Redis DB 혼선
- **위치**: `train-service/.../interceptor/QueueTokenInterceptor.java`
- **원인**: `queue:token:used:{hmacPart}` 키를 **train-service Redisson (DB 1)**에 저장. P5 규칙에서 `queue=DB 2`이므로 큐 관련 키가 DB 1에 있는 것은 명명 및 격리 규칙 위반
- **위험**: Redis 모니터링/디버깅 시 DB 1에서 queue 키를 찾게 되는 혼란, 향후 multi-instance 환경에서 DB 격리 정책 혼선
- **수정 방향**: 큐 토큰 used 키를 DB 2(queue 전용)로 이동하거나, `reservation:token:used:` 로 명명 변경하여 train DB 1 소속임을 명확히

### F-09 · `RateLimitFilter` `localCounts` 점진적 메모리 누수 + 순간 rate limit 해제
- **위치**: `api-gateway/.../filter/RateLimitFilter.java`
- **원인**: Redis 장애 fallback 시 `localCounts`(ConcurrentHashMap)가 분(minute)별 키를 누적. `size() > 5_000`일 때 `clear()` 호출 → 순간적으로 모든 IP의 rate limit 카운터 초기화 → 제한 우회 가능
- **위험**: Redis 장애가 길어질수록 메모리 증가, clear 시점에 rate limit 보호 순간 해제
- **수정 방향**: `localCounts` 키에 TTL 개념 도입(분 윈도우가 지난 키 주기적 정리) 또는 Caffeine 캐시 사용

### F-10 · `BruteForceFilter` OAuth2 경로 포함
- **위치**: `api-gateway/.../filter/BruteForceFilter.java` + `RateLimitFilter.java`
- **원인**: `/oauth2/**`, `/login/oauth2/**` 경로가 `resolveBucket`에서 `"global"` 버킷(100/min)으로 분류. BruteForce 실패 카운트도 OAuth 콜백에 적용됨
- **위험**: OAuth 콜백은 브라우저 리다이렉트이므로 IP 기반 rate limit이 정상 사용자를 차단할 수 있음 (provider redirect가 동일 IP에서 여러 번 발생 가능)
- **수정 방향**: `JwtValidationFilter.PERMIT_PATTERNS`처럼 BruteForceFilter / RateLimitFilter에도 OAuth 경로 명시적 bypass 추가

## 🔵 미수정 — Low / 관찰 필요

### F-11 · `RefreshToken.rotate()` — 이전 access token blacklist 미지원
- **위치**: `auth-service/.../service/RefreshTokenService.java`
- **원인**: refresh token rotate 시 이전 access token을 blacklist에 추가하지 않음 → 탈취된 access token이 TTL(30분)까지 유효
- **현재 허용 범위**: access token TTL 30분이 짧아 실용적 위험은 낮음
- **수정 방향**: access token blacklist(`blacklist:at:{jti}`) Redis 키 도입 또는 JWT short TTL 정책 명문화

### F-12 · `BaseTimeEntity` `createdAt`/`updatedAt` — DB 저장 시 timezone 미지정
- **위치**: `common-lib/.../entity/BaseTimeEntity.java`
- **원인**: `@CreatedDate`, `@LastModifiedDate` 가 `LocalDateTime`으로 저장. MySQL 컬럼 타입이 `DATETIME`이면 timezone 변환 없이 저장됨 → 서버 timezone 변경 시 기존 데이터와 불일치
- **위험**: 현재 Docker UTC 환경에서는 무방. 서버 timezone 변경 또는 `TIMESTAMP` 컬럼으로 변경 시 기존 데이터 시각 오염
- **수정 방향**: MySQL 스키마를 `DATETIME`→`TIMESTAMP` 또는 서버 timezone 고정 명문화

### F-13 · `ReservationCreatedEvent` 알림 payload — expiresAt 표시 포맷 미정의
- **위치**: `notification-service/.../service/NotificationService.java` → `dispatch()` payload
- **원인**: `expiresAt` 문자열이 `LocalDateTime.toString()` 형태(`"2026-06-02T16:39:02"`)로 알림에 그대로 노출될 수 있음 (F-05와 연결)
- **위험**: 사용자 향 알림에 시스템 내부 포맷 문자열 노출
