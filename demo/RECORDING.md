# 🎬 XRail 데모 촬영 대본 (RECORDING.md)

> 목표: 채용 심사자가 **6초 안에** 핵심 강점 3개를 "화면에서 움직이는 것"으로 판단하게 한다.
> 총 길이 **2~3분**. 터미널 + 브라우저(Grafana/Zipkin) 화면만 사용. 코드 에디터는 비춘다면 잠깐만.

## 0. 촬영 전 준비 (녹화 X)

```bash
./demo-up.sh            # 전체 스택 + 관측(Prometheus/Grafana/Zipkin), 전부 green까지 대기 (~2-4분)
./demo/seed.sh          # 시드/리셋(멱등). 계정 1,000개 시드 포함 (SEED_USERS=0 이면 skip)
```

- 터미널 폰트 크게(18pt+), 창 넓게. 색상 로그가 잘 보이게 다크 테마.
- 브라우저 탭 3개 미리 열기: Grafana `http://localhost:3000` (대시보드), Zipkin `http://localhost:9411`, (선택) `http://localhost:5173` 프론트.
- 녹화 해상도 1920×1080, 자막은 하단 중앙.

## 🎬 Scene 1 — 동시성 증명 (오버부킹 0)  · ⏱ ~35초

| 항목 | 내용 |
|------|------|
| **화면** | 풀스크린 터미널. 명령 입력 → 표 형태 결과가 또렷하게 출력. |
| **명령** | `./demo/01-oversell.sh` |
| **자막** | "동일 좌석 100명 동시 예매 → 성공은 **정확히 1건**, 나머지 99건 409. 오버부킹 0." |
| **머니샷** | 결과 표: `201 = 1 / 409 = 99 / DB 활성 티켓 1 / 정합성 ✅ PASS (오버부킹 0건)` |
| **나레이션** | "Redis Lua 비트마스크 원자 락 + DB 비관적 락 더블체크. 100 스레드가 동시에 같은 좌석을 노려도 단 하나만 통과합니다." |

**타임라인**
- 0~8s: 명령 실행, "Phase A 큐토큰 100개 발급" 로그가 흐름
- 8~20s: "🚀 동시 예약 100발 발사" → 집계
- 20~35s: **결과 표 클로즈업**(정지 3초). ✅ PASS 강조.

## 🎬 Scene 2 — 대기열 SSE 실시간 순번 감소  · ⏱ ~45초

| 항목 | 내용 |
|------|------|
| **화면** | 터미널. 순번 숫자가 한 줄에서 **실시간으로 줄어드는** 애니메이션(3초마다 100씩). |
| **명령** | `./demo/02-queue.sh` (촬영을 빠르게 하려면 `QUEUE_USERS=500 ./demo/02-queue.sh`) |
| **자막** | "1,000명 대기 → EventSource(SSE)로 순번이 3초마다 100명씩 실시간 감소 → 입장." |
| **머니샷** | `⏳ 내 순번 812 / 대기 1000` → `... 512` → `... 112` → `🎉 입장! ACTIVE` |
| **나레이션** | "Redis Sorted Set 대기열 + 3초/100명 승급 스케줄러. 운영자는 AUTO/FORCE_ON 모드를 실시간 토글합니다." |

**타임라인**
- 0~5s: "운영자 모드 토글 FORCE_ON" 로그 → 강제 대기 전환
- 5~15s: "1,000명 등록" 진행
- 15~40s: **SSE 순번 카운트다운 클로즈업**(핵심 머니샷) — 숫자가 눈에 띄게 줄어듦
- 40~45s: `🎉 입장!` → "AUTO 복귀" 로그

> 팁: 프론트(`:5173`)에 대기열 화면이 있으면 터미널과 나란히 배치해 SSE가 UI에서도 줄어드는 걸 함께 보여주면 임팩트 ↑.

## 🎬 Scene 3 — 부하 테스트 회복 (Grafana/Zipkin)  · ⏱ ~50초

| 항목 | 내용 |
|------|------|
| **화면** | 터미널에서 부하 시작 → **Grafana 대시보드**로 전환(실시간 그래프). 마지막에 Zipkin. |
| **명령** | `./demo/03-load.sh` (백그라운드로 부하 유지하며 브라우저 전환) |
| **자막** | "1,000 동접 부하 — 에러율 **0%**, 예약 p95 **14ms**, HikariCP 안정. 6개 서비스 분산 트레이싱." |
| **머니샷** | Grafana: 에러율 0% 라인 · 예약 p95 그래프 · Hikari active 안정 구간 / Zipkin: 서비스 목록 6개 + gateway→train trace |
| **나레이션** | "커넥션 풀 고갈·서킷 오픈을 잡아 1,000 동접에서 에러율 0을 달성했습니다." |

**타임라인**
- 0~10s: `./demo/03-load.sh` 실행, "본 테스트 1,000 동접" 로그
- 10~35s: **Grafana 화면**(`localhost:3000` → XRail 대시보드). "예약 생성 TPS", "p95", "HikariCP active" 패널을 순서대로 훑기
- 35~48s: **Zipkin**(`localhost:9411`) 서비스 드롭다운에 6개 서비스 노출 → api-gateway trace 1건 클릭 → gateway→train 스팬 펼침
  (사가의 Kafka 구간은 Transactional Outbox가 별도 스레드에서 발행하므로 단일 trace로 이어지지 않음 — 이벤트의 `traceId` 필드로 로그 상관관계 추적)
- 48~50s: JMeter HTML 리포트 요약(`results/html-demo/index.html`) 에러율 0% 정지 컷

## 실패 시 대처 (촬영 중 문제)

- 스크립트가 어디서 막혔는지 각 단계 `▶ / ✓ / ✗` 로그로 표시됨. `✗` 뒤 안내대로 조치.
- Scene 1에서 `201`이 0이면: `./demo/seed.sh` 재실행 후 재시도(좌석 락/예약 리셋).
- Scene 3에서 `503/No servers available`: 기동 직후 LB 예열 미완 → 30s 후 재시도(스모크 먼저).
